Chapter 01 · The map

A language model is a next-word guessing machine.

That's the whole secret. Everything else — Claude, ChatGPT, the model you're about to build in PHP — is engineering to make that one guess very, very good.

Try it — you are now the language model
Pick the next word — or let chance pick, weighted by probability.

Those bars are a probability distribution: one score per candidate word, all summing to 100%. An LLM produces exactly this — nothing more — then a word is drawn, appended, and the machine guesses again. One word at a time, forever. That loop is called sampling, and you'll build all of it.

§1.1 The one big idea

When you talk to Claude, it feels like a mind. Under the hood there is a single mathematical function. It takes a sequence of words (really tokens — chapter 2) and returns one probability for every word it knows, answering: "what comes next?"

The astonishing discovery of the last decade: if you make that function big enough and train it on enough text, being good at "what comes next" requires understanding grammar, facts, style, and reasoning. To predict the next word of "The capital of Denmark is", you must know it's Copenhagen. To predict the next line of a program, you must model what the code does. Intelligence emerges as a side effect of compression: the best possible next-word guesser has no choice but to understand.

Hold on to this The function never "decides to say" anything. It only scores candidates. Chat behavior, helpfulness, refusals — all of that is trained-in later (chapters 10–11), on top of the same guessing machine.

§1.3 A sense of scale

A model's capacity lives in its parameters — the learned numbers inside it (you'll meet them as weights in chapter 4). The bars below are on a logarithmic scale: each step rightward is 10×.

Our PHP GPT~0.5 M
GPT-2 (2019)124 M
GPT-3 (2020)175 B
Frontier (2026)~10¹²

Same architecture up and down this chart. The difference between our model and a frontier model is almost entirely scale — parameters, data, and the compute to train them.

Which is why we build in PHP without guilt: the concepts are identical at 500 thousand parameters and 1 trillion. What scale buys is coherence — our model will learn spelling and local grammar; frontier models, trained with roughly a trillion times more compute, learn world knowledge and reasoning. In today's mission you'll measure exactly how many multiply-adds per second your Mac can do in PHP, and we'll carry that number all the way to chapter 11 to compute how long training Claude would take on it. §1.5 The build

Chapter 1's code is matrix multiplication — the workhorse of deep learning. By chapter 8, ~95% of the model's runtime will be spent inside this one function. (The ML world calls it matmul; in our codebase it's spelled out as Matrix::multiply().)

The files

llm/
├── src/Matrix.php    ← Matrix::multiply() — read this one
├── bin/bench.php     benchmark: measures your Mac's MFLOP/s
├── bin/phpj         runs php with the JIT compiler on
└── tests/check-ch1.php verifies multiply() is correct

The math (all of it)

Multiplying an n×k matrix A by a k×m matrix B gives an n×m matrix C, where each cell is a row-times-column dot product:

C[i][j] = A[i][0]·B[0][j] + A[i][1]·B[1][j] + … + A[i][k-1]·B[k-1][j]

Concretely: [[1,2],[3,4]] · [[5,6],[7,8]] → cell (0,0) is 1·5 + 2·7 = 19. Matrices in our project are plain PHP arrays of rows: [[1.0, 2.0], [3.0, 4.0]].

The implementation is three nested loops — over rows of the left matrix, over columns of the right, and an inner loop accumulating the dot product. That's the entire function. One detail worth noticing in src/Matrix.php: the current row of the left matrix is copied to a local variable before the inner loops ($leftRow), saving one array lookup per innermost step — in the hottest loop of the whole project, that alone was worth ~25% speed.

Run it

# 1. correctness — four ✓ checks
php tests/check-ch1.php

# 2. benchmark, plain PHP interpreter
php bin/bench.php

# 3. benchmark again with the JIT compiler
./bin/phpj bin/bench.php
Result — measured on this Mac
All checks pass. Plain PHP: 184 MFLOP/s. With JIT: 1,217 MFLOP/s — a 6.6× speedup for two command-line flags. For perspective, one H100 GPU does ~10¹⁵ FLOP/s: about a million times more. That ratio is the entire story of why our models stay tiny, and we'll come back to it in chapter 11.
Why JIT matters here Plain PHP re-interprets every += and array access, billions of times. The JIT compiles your hot loops to native machine code. It's the difference between a model that trains over lunch and one that trains overnight — for free.

Chapter 02 · Tokenization

Before it can read, text must become numbers.

A neural network is a pile of arithmetic. It cannot see letters — only numbers. The tokenizer is the bridge, and it shapes everything the model can and cannot do.

Live tokenizer — type anything (vocabulary: a–z + newline, learned from 32,033 names)

§2.1 A reversible bridge

A tokenizer is two functions that must be perfect inverses: encode() maps text to a list of integer token IDs, and decode() maps IDs back to the exact original text. The set of pieces it knows is the vocabulary, and each piece's position in that list is its ID. That's all a tokenizer is.

Ours is character-level: scanning the names corpus finds 27 distinct characters (a–z plus the newline between names), sorted so IDs are stable: \n=0, a=1 … z=26. So encode("mathias") → [13, 1, 20, 8, 9, 1, 19] — and from here on, the model will never see anything else.

Token 0 is special The newline separating names becomes an ordinary token — and that quietly gives the model a way to say "this name is finished." In chapter 10 the same trick, with tokens like <|user|> and <|assistant|>, is what turns a text-completer into a chatbot. Special tokens are just tokens.

§2.2 Growing a vocabulary: byte-pair encoding

Characters are a tiny vocabulary (27) but long sequences — and compute scales with sequence length. Real tokenizers spend a bigger vocabulary to buy shorter sequences, and they choose the vocabulary from the data itself with an algorithm from 1994: byte-pair encoding (BPE). One rule, repeated: find the most frequent adjacent pair of tokens, glue it into a new token. Watch it run on real names:

Each click performs one merge: the most frequent pair becomes a single new token.

Run against all 32,033 names, 40 merges shorten the average name from 6.12 tokens to 4.39 — 28% less sequence to compute. Frontier tokenizers push the same lever to ~100,000–200,000 tokens (built from raw bytes, so any input tokenizes — no out-of-vocabulary errors), producing tokens like " the", "ing", even whole common words.

§2.3 Tokenization explains LLM quirks

Once merged, a token is atomic — the model receives ID 3,204, not the letters inside it. This single fact explains a family of famous LLM failures:

Counting letters. Ask a model how many r's are in "strawberry" and it must reason about characters it literally cannot see — straw+berry arrives as two opaque IDs. It only knows the spelling if training text happened to discuss it.

Arithmetic. "1234" might tokenize as 12+34 one time and 123+4 another — the model has to learn math over inconsistent chunks.

Cost and context. API pricing, context windows, generation speed — all counted in tokens, because tokens are the true unit the model computes over. (Danish costs more tokens per sentence than English: fewer Danish merges made it into the vocabulary.)
